# Задача для Codex: добавить GitHub-режим для bookmarklet-ссылок

Нужно доработать текущую сборку и `index.html`, не ломая существующую функциональность.

Сейчас `index.html` собирает обычные bookmarklet-ссылки, где исполняемый JavaScript код находится прямо внутри `javascript:` URL. Этот режим нужно сохранить полностью.

Дополнительно нужно добавить второй режим: GitHub loader mode.

## Главная цель

На странице `index.html` должен появиться переключатель режима ссылок:

```txt
javascript / github
```

- `javascript` — текущий режим, работает как сейчас: полный код букмарклета находится внутри `javascript:` ссылки;
- `github` — новый режим: ссылка содержит только короткий bootstrap-loader, который подключает соответствующий файл из GitHub Pages через `<script src="">`.

Нельзя заменять старый режим новым. Нужно именно добавить режим и сохранить возможность использовать оба варианта.

## Пример GitHub-ссылки

Для файла `editor.js` итоговый `href` в режиме `github` должен быть примерно таким:

```js
javascript:(()=>{const s=document.createElement('script');s.src='https://nikolay-a-baranov.github.io/onliner-bookmarklets/dist/editor.js?v=20260516183045';document.body.append(s)})()
```

## Структура

Желательная структура результата:

```txt
src/
dist/
  editor.js
  toc.js
  proofread.js
  manifest.json
index.html
```

`src/` — текущие читаемые исходники.

`dist/` — собранные исполняемые JS-файлы для GitHub Pages.

`index.html` — витрина букмарклетов с переключателем режима ссылок.

## Что должна делать сборка

Для каждого bookmarklet entry нужно формировать два результата:

1. Текущий `javascript:` URL с полным кодом внутри — для режима `javascript`.
2. Отдельный файл `dist/<name>.js` и короткий loader URL — для режима `github`.

То есть сборка должна:

1. Собрать каждый entry так, как он собирается сейчас для обычного bookmarklet URL.
2. Дополнительно сохранить исполняемый итоговый JS в `dist/<name>.js`.
3. Посчитать hash содержимого каждого `dist/<name>.js`.
4. Прочитать старый `dist/manifest.json`, если он есть.
5. Если hash файла не изменился — сохранить старую `version`.
6. Если hash изменился или файла нет в manifest — записать новую `version` текущей датой/временем сборки.
7. Обновить `dist/manifest.json`.
8. Сгенерировать/обновить `index.html` так, чтобы на странице можно было переключать `href` карточек между режимами `javascript` и `github`.

## Версионирование GitHub-режима

Полноценный semver пока не нужен.

Нужно использовать дату сборки как cache-busting версию, но обновлять ее только если изменился соответствующий `dist/*.js`.

Формат версии:

```txt
YYYYMMDDHHmmss
```

Пример `dist/manifest.json`:

```json
{
  "editor.js": {
    "hash": "...",
    "version": "20260516183045"
  },
  "toc.js": {
    "hash": "...",
    "version": "20260516183510"
  }
}
```

Поведение:

- если `dist/editor.js` после сборки не изменился, версия в GitHub-ссылке для `editor.js` должна остаться прежней;
- если `dist/editor.js` изменился, версия должна обновиться;
- `Date.now()` внутри публичного GitHub-loader использовать нельзя;
- `dist/manifest.json` можно коммитить, чтобы версии были стабильны между сборками.

## Поведение `index.html`

На странице нужен явный переключатель режима:

```txt
Режим ссылок: javascript / github
```

Требования:

- по умолчанию можно оставить текущий режим `javascript`, чтобы ничего не сломать;
- при выборе `github` все карточки/ссылки должны получать `href` с GitHub loader;
- при возврате в `javascript` все карточки/ссылки должны получать текущий полный bookmarklet URL;
- переключение должно быть локальным и предсказуемым;
- желательно сохранять выбранный режим в `localStorage`, но без усложнения архитектуры;
- текст/подсказка на странице должны давать понять, что `github`-режим требует интернета и доступности GitHub Pages.

## Конфигурация GitHub Pages URL

Базовый URL нужно вынести в одно явное место, например:

```js
const githubBaseUrl = "https://nikolay-a-baranov.github.io/onliner-bookmarklets/dist";
```

Не размазывать домен по коду.

Если в проекте уже есть конфиг для bookmarklet entries, лучше расширить его, а не создавать параллельную структуру.

## Важные ограничения

- Сохранить текущий режим `javascript` без регрессий.
- Не удалять текущую механику генерации обычных bookmarklet-ссылок.
- Новый GitHub-режим должен быть добавочным.
- Не хранить весь код букмарклета внутри URL в режиме `github`.
- В GitHub-режиме URL должен содержать только loader.
- Основной код должен исполняться из `dist/*.js`.
- Не использовать `Date.now()` в публичных GitHub-ссылках.
- Не обновлять версию при каждой сборке, если итоговый файл не изменился.
- Не делать широкий рефакторинг без необходимости.
- Сохранять текущую структуру проекта, если она уже работает.
- Следовать правилам из `JAVASCRIPT.md`.

## Архитектурное ожидание

Лучше всего сделать один источник данных для карточек/букмарклетов, где для каждого entry известны:

- id;
- title;
- entry/source;
- dist filename;
- обычный javascript href;
- github href.

`index.html` должен только отображать карточки и переключать уже подготовленные значения, а не пересобирать код на клиенте.

Если сейчас генерация устроена иначе, изменить минимально: не переписывать всю систему, а добавить недостающий слой для GitHub href рядом с текущим href.
